# 2026-07-21 밤 — 아이디어 8종 실행 계획 + 어텐션 수확 결과

## 0. 중요한 재정리 — Ver8(챔피언) vs Ver11(보류) 스코프 혼동 수정

오늘 세션에서 브레인스토밍한 8개 아이디어 중 **프루닝 관련 항목은 전부 Ver11 전용
개념**(FitPrune 자체가 Ver11에서만 존재 — 챔피언 Ver8은 서빙에 토큰 프루닝을 아예 안 씀)
이었는데, Ver11은 이미 이번 세션에서 "챔피언 대비 −2.7pp, 트랙 종결" 판정을 받았다
(`postmortem_2026-07-21_a100_and_plan.md`). 즉:

- **어텐션 수확 프루닝·선택-주변화(prune-config) TTA·프레임간 변화 스코어**는 전부
  Ver11의 FitPrune을 고치는 이야기라, Ver11 자체를 살려내지 못하면 **리더보드에 직접
  영향이 없다** — 프루닝만 완벽해져도 Ver11의 다른 −2.7pp 원인(Stackelberg LoRA, KT
  손실, 자체 score24 헤드 등)까지 다 해결해야 챔피언을 넘는다. 지금은 **발표용
  독창성·포렌식 연구**로 취급하는 게 맞고, "이걸 고치면 챔피언을 이긴다"는 기대는 하지 말 것.
- **캡션 의존도 진단·위치편향 증류·뷰-일관성 손실**은 챔피언(Ver8) 자체에 바로 적용
  가능 — 이쪽이 실제 순위에 영향을 줄 수 있는 후보다.

아래 §1(오늘 완료)은 전부 Ver11 스코프, §2("A100 즉시 실행 준비")부터는 Ver8 스코프로
재우선순위했다.

## 1. 오늘 완료 — Ver11 프루닝 축 3연속 실험, 전부 약하거나 null

| 실험 | 결과 | 스크립트 |
|---|---|---|
| 유저의 "단위별 분리→병합" 아이디어(이벤트 크라우드아웃 검증) | starvation 0.1~0.4% — 고칠 문제가 거의 없음 | `Ver11/scripts/measure_event_starvation.py` |
| 선택-주변화 TTA 사전판정(prune-config 간 keep-set 오버랩) | Jaccard 0.62~0.82 — 다양성은 있음(단, Ver11 전용이라 실행 우선순위 낮춤, §0 참고) | `Ver11/scripts/measure_selection_overlap.py` |
| **어텐션 수확 프루닝**(오늘 신규, 아래 상세) | **spearman +0.066, top50%-jaccard 0.351(우연 기준선 0.333 근처) — 약한 신호, 명확한 승리 아님** | `Ver11/scripts/measure_attention_vs_cosine.py` |

### 어텐션 수확 상세 결과 (교훈 포함)

fitprune.py가 자기 코사인 스코어를 "FitPrune 실제 어텐션의 학습-불필요 서로게이트"라
부르는 걸 실제 문맥화 어텐션으로 교체해봤다. 포렌식 샘플 3건(diZi5g·u7w0lr·2vqGOF),
64층 중 21·42층, `attn_implementation="eager"` + 훅.

- **버그 하나 발견·수정**: 캡션 토큰 위치를 별도 재토큰화로 찾으면 안 됨 — 이미지
  플레이스홀더가 실제 멀티모달 프로세서 안에서 이미지당 수백 토큰으로 확장되는데,
  별도 텍스트 토큰화는 이 확장을 모른다. 실제 `input_ids` 안에서 서브시퀀스 검색으로
  고쳐야 함(스크립트에 반영 완료) — 안 그러면 "레이어 두 개가 완전히 같은 값"이라는
  가짜 결과가 나온다(처음에 실제로 이렇게 나왔었음).
- **어텐션 싱크 확인**: position 0(BOS류)이 캡션 토큰 쿼리 전체 어텐션의 **34~65%**를
  독식. 싱크 제외 재정규화는 순위 기반 지표(spearman/jaccard)엔 영향 없음(상수 제거는
  순위를 안 바꿈) — 이미 반영해서 재확인함.
- **최종 판정**: 64개 헤드를 통째로 평균 낸 "순진한" 어텐션 수확은 코사인 서로게이트와
  **거의 우연 수준으로만 일치**(spearman +0.066은 사실상 무상관, jaccard 0.351은 무작위
  50% 겹침 기댓값 0.333에 근접). 이게 "코사인이 이미 맞다"는 뜻도, "실제 어텐션이 별로다"는
  뜻도 아니다 — 둘이 애초에 다른 걸 재는 것일 수도 있고, 64개 헤드 평균이 진짜 의미
  신호를 가진 소수 헤드를 희석했을 수도 있다. **어느 쪽이 맞는지는 다운스트림 정확도로만
  가릴 수 있는데, 그러려면 실제 재학습이 필요 — 그런데 그 재학습의 사전 신호가 이 정도로
  약하면 투자 우선순위가 낮다.**
- **권고**: 이 방향에 A100을 더 쓰지 말 것. 특정 헤드 선별(head-probing) 같은 정교한
  후속이 있으면 이득이 있을 수 있으나, Ver11 자체가 보류 상태라 지금 급하지 않음.

## 2. Ver8(챔피언) 적용 가능 — 실제 순위에 영향 줄 수 있는 것들

### ✅ 2-1. 캡션 의존도 진단 — A100 즉시 실행 준비 완료 (TTA8 균형 끝나면 여기 태울 것)

**`Ver8/scripts/measure_caption_dependence.py` 작성 완료, import·데이터로딩 로직
드라이런 검증 완료**(GPU 없이 — 발열 이슈로 실제 forward는 로컬에서 안 돌림).
train 200건을 {실제 캡션 / 빈 캡션 / 다른 샘플 캡션으로 스왑} 3조건으로 채점해 정확도
차이를 본다 — 학습 없음, 모델 무변경, 순수 측정.

```bash
cd Ver8 && export PYTHONPATH=$PWD/src && export LD_LIBRARY_PATH=~/anaconda3/envs/py3_11/lib
python scripts/measure_caption_dependence.py \
    --csv data/train.csv --image-dir data/train \
    --adapter "runs/DPO-checkpoint-600" --n 200 --out runs/caption_dependence
```

예상 소요: 200건×3조건=600회 score24 forward, TTA 없이 단일뷰라 test 819건 서빙(TTA4
기준 ~1h)보다 가벼움 — A100에서 수십 분 예상. 산출물: `runs/caption_dependence/summary.json`
(3조건 EM), `progress.jsonl`(재개 가능, 샘플별 상세). **캐비엇(스크립트 자체에도 명시)**:
train 전량학습이라 암기가 캡션 없이도 정답을 낼 수 있어 의존도가 과소측정될 수 있음 —
보고 시 반드시 언급.

### 2-2. 위치편향 증류로 TTA 뷰 감축 — 코드 변경 필요, 계획만

**주의 수정**: Agent B가 이 아이디어를 "확정된 25.7GiB/24GB OOM 문제 직결"이라고
설명했는데, 그 OOM은 **Ver11의 stage2=always+FitPrune 조합에서만** 확인된 것 — 챔피언
Ver8의 TTA3~8은 전부 4090에서 이미 정상 완주했다(`champ_tta_sweep.log`). 즉 이 아이디어는
"OOM 해결"이 아니라 순수 **서빙 비용 절감**(뷰 수↓) 스토리로 봐야 함 — 여전히 발표의
"자원 효율" 축엔 유효하지만, 급한 문제 해결은 아니다.

실행 계획(코드 변경 선행 필요, 아직 안 함):
1. `Ver8/src/snuai/infer/tta.py` 또는 `predict.py`에 뷰별 24점수 로그를 `progress.jsonl`에
   같이 저장하는 계측 추가(몇 줄).
2. train 300건을 TTA4(4뷰)로 돌려 뷰별 로그확률 확보.
3. 오프라인으로 위치편향 벡터(24차원 또는 4×4 슬롯별) 적합 — 2-fold 교차검증.
4. `(1~2뷰 + 편향보정)` EM vs `(4뷰 평균)` EM 비교. 덤으로 train 라벨 24클래스 히스토그램도
   확인(1분, 비균등이면 log-prior 보정 공짜).
5. 게이트: 편향보정 EM이 4뷰 대비 유의하게 안 떨어지면 서빙 비용 절감안으로 채택.

### 2-3. 뷰-일관성 정규화 — 학습 슬롯 필요, 계획만

같은 샘플을 두 뷰(σ)로 forward해 원공간 정렬 후 score24 분포 간 대칭 KL을 보조손실로.
**A100 훈련 슬롯이 있어야 함** — 지금 Ver13이 그 A100을 쓰고 있으므로 당장은 대기.
사전 확인 가능한 것(학습 없이): 챔피언의 Klein TTA4 train-set 뷰간 불일치율을 재보면
이 손실의 이득 상한을 추정할 수 있음(2-2의 계측을 그대로 재사용) — 2-2를 먼저 하면
공짜로 딸려온다.

### 2-4. 캡션 시간접속사 서수 마크업 — 진단 선행 필요

프롬프트 SSOT(`prompting.py`) 변경 + 재학습 동반이라 가장 무거움. **2-1(캡션 의존도
진단)에서 "접속사 있는 캡션의 오답률이 실제로 높은지"부터 확인한 뒤에만 투자 가치 판단**
— 순서상 2-1 다음.
