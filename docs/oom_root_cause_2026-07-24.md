# OOM 반복 발생 근본 원인 (2026-07-24)

## 증상
24GB 카드(RTX 4090)에서 TTA8 balanced + motion_weight + stage2=always 조합으로 test819
추론을 돌리면, 대략 test.csv **578번째 행(0-indexed)부터** CUDA OOM이 반복 발생한다
(motion03: 2026-07-23, boost_frac=0.5: 2026-07-24, 둘 다 서로 다른 서빙 설정인데도 같은
지점에서 시작 — `scripts/run_pre_supervised.sh` 주석의 2026-07-22 최초 관측(index 728)도
같은 구간).

## 원인
프로세스 수명 누적 버그가 아니라 **test.csv 자체의 캡션 길이 분포가 균일하지 않음**:

| 구간 | 캡션 길이 평균 |
|---|---|
| 0~577행 | 154자 |
| 578~819행 | 459자 (약 3배) |

578행에서 계단식으로 전환되어 끝까지 유지된다(윈도우별 평균: `[550:600]=276.6`(전환 구간),
`[600:650]=472.1`, 이후 끝까지 450~472 유지). 캡션이 길수록 FitPrune 크로스모달 앵커 토큰과
LLM에 들어가는 텍스트 프롬프트 토큰이 늘어나고, 이미 비전 토큰만으로 VRAM 한계에 가까운
TTA8+stage2=always 조합에서 그 증분이 임계점을 넘긴다.

재현/검증 방법:
```python
from snuai11.data import load_samples
samples = load_samples("data", "test")
lens = [len(s.caption) for s in samples]
# lens[:578] 평균 154, lens[578:] 평균 459
```

## 왜 이게 "프로세스 누적"으로 오인되기 쉬웠나
- 크래시가 특정 절대 인덱스(578~) 이후로 몰려서, 마치 "오래 돌수록 나빠진다"처럼 보인다.
- 실제로는 test.csv 순회 순서(무작위 셔플 없음, `data.py load_samples`가 csv 행 순서 그대로)가
  우연히 "짧은 캡션 구간 → 긴 캡션 구간"으로 나뉘어 있어서, 순서대로 처리하면 후반부에서만
  집중적으로 무거운 샘플을 만나는 것처럼 보인다.
- `torch.cuda.empty_cache()`+`gc.collect()`로도 안 풀리는 이유도 이걸로 설명됨 — 프래그먼트가
  아니라 그 샘플 자체가 진짜로 더 많은 토큰을 요구하는 것이라, 캐시를 비워도 필요한 양은
  줄지 않는다.

## 대응
- 이미 넣은 방어책(in-process 재시도 1회 + 프로세스 재시작 supervisor)은 여전히 유효 —
  근본 원인이 "그 샘플이 진짜 무겁다"는 것이므로, 서빙 설정(TTA8/stage2=always)을 유지하는
  한 완전 제거는 어렵고, 결국 그 무거운 샘플들에서 이따금 재시도/재시작이 필요하다.

  훨씬 빠듯하다"는 점을 자원 효율성/최적화 방법론 서술에 정직하게 반영할 것 — 이 진단 자체가
  root-cause 분석 근거로 쓸 수 있다.
