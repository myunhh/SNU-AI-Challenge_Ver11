  순열 중 정답 하나를 맞히는 24-way 분류 문제). SNU AI Challenge 2026(서울대 데이터사이언스대학원),
  Kaggle 플랫폼 진행. 예선 마감 **2026-07-24 23:59**, Kaggle 제출 1일 2회 제한.
- **환경 제약(규정)**: 모델은 인터넷 차단 로컬(CPU/GPU) 환경에서 실행 가능해야 함. 외부 상용
  API(ChatGPT/Gemini/Grok 등) 학습·추론 사용 금지(전처리 목적 한정 허용, 총액 3만원 이내).

  **모델 앙상블(여러 모델 결과 조합, 또는 같은 모델을 데이터 나눠 파인튜닝 후 조합) 금지** —
  단 TTA(test-time augmentation)는 규정 3.3에 명시적으로 허용됨. 테스트셋 전체 추론이 검증

## 2. Ver11 아키텍처 — 세 개의 독립된 축

Ver11의 한 문장 요약: **"같은 32B 모델에게 더 좋은 눈(입력)과 더 좋은 채점표(출력)를 주고,
더 안정적으로 가르친다(학습)."** 세 축이 각각 입력·출력·학습을 담당하며 서로 독립적이라
한 축이 실패해도 나머지가 살아남는 설계. Ver10까지의 코드를 상속하지 않은 독립 구현
(패키지명 `snuai11`, 별도 원격 저장소) — 단 **코드 독립**이지 **가중치 재사용 금지**는
아니어서, warm-start(§3)는 별개 판단으로 채택했다.

### 2-1. 축① Cross-Targeted FitPrune — "캡션이 지목한 것만 본다"

- **문제**: 프레임 4장 고해상도 입력 시 샘플당 비주얼 토큰 ~4,400개 — 32B 모델의 VRAM·속도를
  압박하고, 캡션과 무관한 배경(물/눈/타일)이 순서 판별에 필요한 소형 물체(스키어/공/얼굴)를
  밀어냄.
- **방법**: 캡션을 규칙 기반 4이벤트로 쪼갠 뒤, 각 이미지 토큰이 4개 이벤트 전부와 얼마나
  관련 있는지 교차 채점(4×4)해 상위 **50%**만 남기고 LLM 진입 전에 컷. 잔류 토큰 선택 기준은
  **중요도 80% + 다양성 20%**(max-min greedy MMR).
- **논문 근거**: FitPrune(AAAI'25)의 "텍스트-조건부(cross) 신호가 self 신호보다 우월"
  실험(GQA 60.1 vs 55.4)을 채택, 50% keep을 안전 지대로 사용. 레이어별 스케줄·이진탐색·
  in-LLM attention 통계는 전부 버리고, **pre-LLM 단일 컷 + 임베딩 공간 행렬곱 1회**로
  경량화(SDPA 유지 = 추론 속도 손실 없음).
- **실제로 겪은 함정 2건과 수정** (정성 검증을 위한 `runs/prune_viz` 렌더링에서 발견 —
  "정량 지표만 보지 말고 중간 산출물을 눈으로 볼 것"이라는 Ver2 사고 이후 정립된 습관이 여기서도 효과):
  1. **rank-1 퇴화**: 4개 이벤트의 중요도 맵이 사실상 동일해지는 문제 — 원인은 이미지/텍스트
     모달리티 간 임베딩 anisotropy가 신호를 압도. **모달리티별 자기-평균 센터링**으로 해결
     (32B 임베드 400캡션 실측: 이벤트 쌍코사인 +0.20(raw) → −0.33(수정 후)).
  2. **stuff-over-things**: 캡션 장면명사(pool/ice/snow)와 매칭되는 대면적 배경이 top-k 예산을
     독식하고 정작 순서 판별용 소형 전경 물체가 잘림. **객체성 블렌드**(`objectness_weight=0.3`,
     센트로이드-residual norm을 코사인과 블렌드) + **MMR 선택**(`mmr_lambda=0.5`, 중복 배경을
     소수 대표로 압축)으로 해결. λ 하한은 최악 케이스 구제 조건에서 수학적으로 유도(λ>0.4).

### 2-2. 축② score24 헤드 — "24개 중 고르게 한다"

- **문제**: 일반 VLM은 답을 글자로 생성 — 느리고 형식 붕괴 위험, 확률을 다루기 불편.
- **방법**: 모델 마지막에 `Linear(5120→24)`를 붙여 24개 순열 후보에 **1회 forward**로 점수를
  매김(one-pass, 생성 없음). 초기값은 기존 `lm_head`의 A~X(24글자) 행을 그대로 가져옴
  (`init_from_lm_head`) — 학습 시작 전부터 이미 warm-start 베이스(Ver8)와 같은 채점 논리를 재현.
- **장점**: ① 생성 없어 빠름 ② 24개 확률이 정직해 **마진**(1위-2위 점수차)을 TTA 집계·
  stage-2 캐스케이드 트리거·boost_frac 선택 기준으로 재사용 가능 ③ 형식 붕괴가 구조적으로 없음.

### 2-3. 축③ Stackelberg 두-시간축 QLoRA — "몸통과 머리를 다른 속도로 가르친다"

- **문제**: 새로 붙인 헤드(②)는 백지 상태, 몸통(LoRA)은 이미 언어·시각 능력을 갖춤 — 같은
  학습률로 가르치면 헤드가 못 따라오거나 몸통이 망가짐.
- **방법**: 헤드를 몸통보다 **5배 빠른 학습률**로(게임이론 Stackelberg 리더-팔로워 구조,
  몸통=리더/헤드=팔로워). **헤드 weight decay 0.1은 장식이 아니라 이론상 강볼록성 수렴
  조건**. LoRA r16/α32/dropout0.05, 언어 레이어 7개 projection(q,k,v,o,gate,up,down) full-path.

## 3. 학습 방법론 — 실패와 수정의 기록

### 3-1. 1차 본런 실패 → warm-start 채택 (2026-07-16)

- 무작위 초기화 LoRA에서 2000스텝 SFT 본런 시, loss가 **ln(24)≈3.178**(24클래스 균등분포
  엔트로피 = "아직 아무것도 못 배우고 찍는 상태") 근방에 정체되다 코사인 스케줄의 약 65%
  지점(step 1260~1290)에야 탈출 — **A100 비용의 2/3을 랜덤워크에 태운 셈**.
- **원인 진단**: Stackelberg 헤드 LR(peak, body의 5배 ≈1e-3)이 작은 유효배치(`accum=4`, 스텝당
  4샘플)의 그래디언트 노이즈 대비 과도해 초반 다수 스텝이 순학습이 아닌 랜덤워크에 가까웠음.
- **기각한 대안**: `--schedule poly`(초반 감쇠가 빨라 peak 체류를 줄일 것으로 기대) — 정밀
  재분석 결과 poly는 총 LR 예산을 cosine 대비 body 19.9배/head 6.3배나 깎아, 그대로 본런에
  썼으면 이번엔 under-training으로 실패했을 것으로 판명(진단런 조합을 본런에 그대로 옮기면
  안 된다는 교훈 자체가 기록할 가치).
- **최종 채택 — Ver8 챔피언 어댑터(DPO checkpoint-600) warm-start**: 밑바닥 대신 이미 LB
  0.91099를 내는 가중치에서 이어받아 학습 — 플래토 구간을 원천 제거. 호환성 전수 검증(base/
  LoRA 구성/peft 버전 동일), `init_from_lm_head`가 Ver8의 채점 로직을 정확히 재현해 자연스러운
  전이 지점. 러너에 **10스텝 전이 게이트(평균 ce<3.0)** 내장해 전이 실패를 조기 감지.
  레시피: body-lr 5e-5, accum 16(4→16, 유효배치 확대로 그래디언트 노이즈 직접 감소),
  cosine, 1500스텝.
- **부수 효과 — DPO 기하 보존을 위한 기대-Kendall 보조손실**: Ver8의 DPO는 인접-스와프
  hard-negative로 "오답들 사이의 거리 구조"를 학습시켰는데, 그 위에 순수 CE SFT를 얹으면
  이 구조가 씻겨나갈 위험. `loss += 0.5·E_p[KendallTau(pred,GT)/6]`(`--kt-weight 0.5`)를 추가
  — 정답이 one-hot이면 이 항이 정확히 0이라 CE 최적점과 충돌 없이, 틀렸을 때 "얼마나 크게
  틀렸는지"를 벌점화. LB가 쌍순서 계열이라는 정황(Ver7 포렌식, 이후 EM 단독으로 확정됨)과도
  방향이 맞음.

**규정 관점**: warm-start는 앙상블이 아니다 — 최종적으로 단일 모델 하나의 가중치를 만들 뿐
추론 시 결합이 없고, 외부 데이터·외부 모델도 개입하지 않는다(제공된 train으로 직접 학습한
어댑터를 이어받는 것). 코드 독립성 요구(Ver10까지 코드 미상속)와는 직교하는 결정.

### 3-2. DPO 재도입 시도 — 두 차례 모두 역효과로 확인 (07-21, 07-23~24)

Ver8(No Prune SFT → DPO 학습, 즉 FitPrune 없이 표준 SFT 후 DPO를 얹는 파이프라인)에서는
DPO(인접스와프 hard-negative)가 성공 사례(0.90226→0.90401)였기 때문에, 같은 아이디어를
Ver11 SFT 위에도 얹어봤으나 **두 번 모두 성능 향상으로 이어지지 않았다** — 우연이 아니라
구조적 문제로 판단.

| 라운드 | 베이스 | 판정 |
|---|---|---|
| 07-21 | ws8-adapter_final 위 600스텝 | 사전 검증에서 개선 미확인, 재검증 후 제외 |
| 07-23~24 | ckpt1200(motion_weight=0.3 학습) 위 1000스텝 | 사전 검증에서 SFT-only 대비 열세 확인, 제출에서 제외 |

- **가설(미검증이나 두 라운드 결과와 정합)**: kt-aux 보조손실(§3-1)이 이미 쌍순서 정렬을
  타겟팅하고 있는데, 그 위에 DPO의 인접스왑 마진 손실을 또 얹으면 같은 목적을 과교정하는
  것으로 추정. Ver8은 이 보조손실 없이 순수 CE SFT 위에서 DPO가 작동했다는 점에서 전제가 다름.
- **07-24 확인한 부가 사실**: DPO 600/800/1000 세 체크포인트를 train 데이터 일부로 비교했더니
  **예측이 서로 완전히 동일**(argmax가 전혀 안 바뀜) — DPO의 margin loss
  (`-log sigmoid(β·margin)`)가 이미 정답인 샘플의 마진만 계속 넓힐 뿐 결정 경계를 못 넘는
  구조(loss는 계속 감소해도 정확도는 이미 천장에 닿아 있던 것). 코드 레벨로 정확히 설명되는
  현상이라 "체크포인트 스텝 스윕이 무의미하다"는 이 프로젝트의 기존 교훈(Ver8에서도 동일
  관측)을 재확인.
- **결론**: 이 트랙에서 "kt-aux SFT + 그 위 DPO" 조합은 구조적으로 안 맞는다고 판단, DPO
  체크포인트 전부 제출 후보에서 제외.

### 3-3. 모션 신호 서브트랙 (07-22 착수)

- **가설의 근거**: 챔피언(Ver8) 쪽 캡션 의존도 진단에서 **이미지만으로도 정답률이 랜덤의
  5배**로 나온 것을 계기로, 캡션-코사인 신호와 상보적인 프레임 간 물리적 모션 신호를 FitPrune
  스코어링에 블렌드하는 아이디어를 설계.
- **사전 검증(저비용)**: 모션 점수가 객체성과 spearman +0.693(강한 정합), 기존 캡션-코사인
  신호와는 +0.011(거의 독립) — 상보적 신호라는 가설을 지지.
- **구현**: `motion_weight` 파라미터(기본 0=기존과 동일, ablation 정확 — 0이면 기존 파이프라인과
  완전히 동일한 코드 경로). 학습 시 `motion_weight=0.3`으로 적용.

  일관 전략 — 배경 노이즈를 걸러내고 캡션이 지목한 신호에 모델 용량을 집중시킴.

  노출시켜, 추론 시 pruned+full 두 단계(§5) 모두에서 잘 작동하도록 함.
- **캡션 이벤트 분할 재설계 검토·기각**(07-16): "16변형 per-event 프루닝" 아이디어를 검토했으나
  train 캡션 9,535건 중 94%가 자연 절(clause) 4개 미만(1절 35%/2절 45%)이라 "4이벤트=4프레임
  1:1 배정" 전제가 성립하지 않아 S1 데이터 게이트에서 기각. 현행 max-pool(이벤트 무관 union)
  방식이 데이터 특성상 더 강건함을 확인한 사례.

## 5. 최적화 방법론 (추론 환경 제약에 맞춘 성능 최적화)

| 기법 | 내용 | 효과 |
|---|---|---|
| FitPrune 50% 토큰 컷 | §2-1 | VRAM·속도 절감, pruned 0.41s vs full 0.57s/forward(로컬 검증) |
| 4-bit 양자화(32B 베이스) | unsloth bnb-4bit | 24GB 카드에 32B를 태우는 유일한 방법(bf16은 VRAM 초과) — Ver4에서 이미 "4bit 서빙은 32B에서만 예외 허용"으로 확립(8B는 반드시 bf16, LB 실측 ~4pp 페널티 확인됨) |
| 균형 TTA8 (D4 코셋) | Klein 4원군을 8뷰로 확장, 각 입력이 각 슬롯을 정확히 1회씩 방문(위치편향 정확 상쇄) | 이 프로젝트 전체 서빙 변경 중 최대 단일 증분(TTA5→TTA8 실측 +1.22pp, 챔피언 트랙) |
| stage-2 always (풀토큰 2단계) | 저마진 캐스케이드 대신 전 샘플에 풀토큰 재forward 집계 | 24h 예산 대비 거의 무료(비전 인코딩 재계산 없이 LLM 재forward만) — 프루닝이 버린 증거를 되살림 |
| boost_frac (2026-07-23~24, 진행 중) | 이미 pruning으로 살아남은 top-scoring 토큰을 LLM 입력 시퀀스에 추가 복제(`boost_copies`회) — 원본/pruned 토큰을 섞는 게 아니라 **이미 학습된 분포 내에서** 상위 토큰의 attention 비중만 재조정 | 검증 중(§7) |
| in-process OOM 복원력 | `predict_sample_resilient` — 캐시 비우기+짧은 재시도 후, 그래도 안 되면 프로세스 재시작으로 위임(2단 방어) | 24h 완주 보장, §6 참고 |

**규정 준수 확인**: FitPrune 두 단계(pruned pass + full pass) 및 TTA8 다중 뷰는 전부 **동일한
단일 가중치 세트**로 입력만 바꿔 여러 번 forward하고 출력 확률만 앙상블하는 방식 —
규정이 명시적으로 허용하는 test-time augmentation 범주이며, 금지 대상인 "여러 모델(또는
여러 파인튜닝 체크포인트)의 결과 조합"과는 무관함을 확인.

## 6. 자원 효율성 — VRAM 제약과의 씨름 (2026-07-23~24 실측 기록)

### 6-1. 실측 VRAM

- RTX 4090(24GB) 로컬 카드 기준, TTA8+motion+stage2=always+boost 조합 순수 추론 피크
  **23.63GiB / 24.56GiB(98.5%)** — 24GB급 카드의 사실상 한계선.
- 과거(TTA3+cascade) 피크는 19.9GiB였던 것과 비교하면, TTA8+stage2=always로 정확도를
  올리는 대가로 메모리 여유가 크게 줄었음 — 정확도-자원 트레이드오프를 정직하게 인지.

### 6-2. OOM 반복 발생 — 근본 원인 진단 (2026-07-24)

814건 완주 과정에서 CUDA OOM이 반복 발생(motion03 런 153회, boost_frac=0.5 런 다수).
처음엔 "장시간 프로세스 실행에 따른 메모리 누적"으로 의심했으나, **정밀 진단 결과 원인은
test.csv 자체의 구조적 특성**임을 확인:

- test.csv **578번째 행(0-indexed)부터 캡션 길이가 평균 3배**(0~577행 평균 154자 →
  578~819행 평균 459자)로 계단식 증가, 끝까지 유지.
- 크래시 시작 지점(≈580번대)이 이 캡션 길이 전환점과 정확히 일치 — 어제(모션03)·오늘
  (boost_frac) 서로 다른 서빙 설정에서도 동일 지점, `run_pre_supervised.sh`의 07-22 최초
  관측(index 728)도 같은 구간.
- 캡션이 길수록 FitPrune 크로스모달 앵커 토큰 + LLM 프롬프트 토큰이 늘어나, 이미 비전
  토큰만으로 한계에 가까운 조합에서 그 증분이 임계점을 넘김. `torch.cuda.empty_cache()`로도
  안 풀리는 이유도 이걸로 설명됨(캐시 정리가 아니라 진짜 더 많은 토큰이 필요한 것).
- **검증 방법**: `load_samples()`로 로드한 캡션 길이의 윈도우 평균을 직접 계산해 확인(상세는
  `docs/oom_root_cause_2026-07-24.md`).

### 6-3. 대응 — 2단 방어 체계 구축

1. **in-process 재시도**(`predict_sample_resilient`, `infer.py`): OOM 발생 시 `gc.collect()`+
   `torch.cuda.empty_cache()` 후 같은 샘플 즉시 재시도(1회, 짧은 대기) — 처음엔 5회+지수
   백오프로 설계했으나, 실측 결과(37회 발동 중 36회가 5회 다 써도 실패) 이 상황에서는 대기가
   순수 낭비임을 확인하고 1회로 축소해 크래시 사이클을 절반 이하로 단축.
2. **프로세스 재시작 supervisor**(`scripts/run_pre_supervised.sh`): in-process 재시도가 다
   실패하면(진짜 필요량이 부족한 경우) 프로세스를 통째로 재시작 — `progress.jsonl` id 기반
   재개라 데이터 손실 없음.
3. **(TODO, 07-28 대비)** 현재 ②는 별도 bash 래퍼라 채점 환경이 `python run_pre.py`를
   직접 호출하면 보호받지 못함 — 재시작 로직 자체를 `run_pre.py`/`infer.py`에 내장할 계획.

## 9. 이 세션에서 발견·수정한 인프라 버그 (재현성 관점에서 기록할 가치)

1. **정전 복구 시 발견 — checkpoint 디렉터리 재사용 시 config drift 무방비**: `--out`을 재사용해
   이전과 다른 하이퍼파라미터(예: `motion_weight`)로 학습을 이어가면, 각 체크포인트의
   `train_args.json`은 개별적으로 정직해도 재사용 자체를 막는 장치가 없어 실수로 다른 설정의
   체크포인트가 같은 디렉터리에 섞일 위험이 있었음. `train_sft.py`에 `check_out_reuse()` 추가.
2. **추론 쪽의 동일 클래스 버그**: `progress.jsonl`이 순수 id 기준으로만 재개되어, 크래시 후
   재시작 시 다른 `--adapter`로 재개해도 막는 장치가 없었음. 실제로 `runs/triage_boost_0.5`가
   DPO 어댑터로 만든 216건과 SFT-only 어댑터로 만든 이후 건들이 한 파일에 섞이는 사고가
   실제로 발생 — id 대조로 오염 제거 후 `infer.py`에 `check_resume_config()`(adapter/head
   포함 드리프트 검사) 추가해 재발 방지.
3. **정전으로 인한 파일 손상 대응**: 진행 중 강제 종료 시 `progress.jsonl` 마지막 줄이 NUL
   바이트로 잘리는 현상 확인 — 손상된 trailing 레코드만 건너뛰고 재개하도록 파서를 방어적으로
   수정.

이 세 가지 모두 "07-28 재현성 검증"을 염두에 두면 단순 버그 수정이 아니라 **제출 코드의
신뢰성을 담보하는 작업**으로 볼 수 있음.
